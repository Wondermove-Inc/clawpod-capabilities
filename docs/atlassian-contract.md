# Atlassian Cloud capability contract

The capability exposes 29 service-namespaced commands over Jira Cloud REST v3, Confluence Cloud REST v2/v1, and agent-local Atlassian OAuth 2.0 (3LO). `command_contracts.json` is generated and is the machine-readable inventory.

All output is a JSON envelope with `ok`, `schemaVersion`, `command`, `requestId`, `effects`, `provenance`, and either `data` or a stable `error` (`code`, `message`, `retryable`, `ambiguousCommit`). Secrets are reference-only and recursively redacted.

Reads have bounded 30-second timeouts and at most three retries for 429/502/503/504, honoring bounded `Retry-After`. Mutations never execute without a one-time, installation-key-bound durable confirmation issued by a successful dry-run, fingerprinted to site, command, and request, and valid for five minutes. Confirmation and idempotency records use locked atomic updates; confirmation tokens use HMAC-SHA256. Network/timeout failure after a mutation is marked ambiguous and must be reconciled before retry. Attachment paths must remain beneath an explicit transfer root and cannot be symlinks. Site aliases bind separate Jira and Confluence HTTPS base URLs plus one credential provider, preventing cross-service or cross-tenant credential selection. Basic auth can derive both service URLs from one site origin; OAuth 2.0 3LO uses service-specific cloudId URLs. Batches contain at most 100 independently validated items, preserve caller input, report indices and a stable partial-failure summary, and stop only on systemic failures.

OAuth onboarding uses one organization-managed distributable confidential 3LO app, an exact fixed loopback callback registered in the Atlassian Developer Console, an unguessable state value, a managed-browser DevTools endpoint, rotating refresh tokens, and atomic mode-0600 credential/site files beneath a transfer root. Atlassian documents authorization-code exchange with a client secret but does not document PKCE for 3LO, so the Harness does not add unsupported PKCE parameters. It always resolves the requested site through `accessible-resources`, uses the cloud ID service origins, and never returns an authorization URL, code, client secret, access token, or refresh token.

Official semantic sources: Atlassian Jira Cloud REST API v3 reference, Confluence Cloud REST API v2 reference, Confluence Cloud REST API v1 search and content attachment references, Atlassian OAuth 2.0 (3LO) app guidance, and Atlassian Cloud rate-limit guidance.
